# ADR-002: Use RS256 JWT for authentication

**Status**: Accepted  
**Date**: 2024-01-01

## Context

The API needs stateless authentication. Options: session cookies, symmetric JWT (HS256), asymmetric JWT (RS256), opaque tokens.

## Decision

Use **RS256** (RSA) signed JWTs with short-lived access tokens (15 min) and rotated refresh tokens stored in Redis.

## Rationale

- RS256 lets future services verify tokens using only the public key — no shared secret.
- Short access token TTL limits the blast radius of a leaked token.
- Refresh token rotation (one-time use) prevents replay attacks.
- Redis refresh token store enables instant revocation.

## Consequences

- Requires key pair generation at setup (`generate-jwt-keys.js`).
- Private key must be kept secret on the backend — never sent to the client.
- Redis is now a hard dependency for auth.
